# Web Runtime Prototype

This example proves the boundary shaped in `tbp-z74`:

- Paint generates backend artifacts.
- The canonical design-system schema sits above those artifacts.
- An authored web projection sits between that schema and the generated web runtime IR.
- A derived `system.web.json` becomes the shared web runtime IR.
- Both a pure custom element and a real Storybook workspace consume the same generated adapter.

The repo-level contract for what those consumers may and may not depend on lives in
[`docs/live_consumer_contract.md`](../../docs/live_consumer_contract.md).

The prototype is intentionally narrow:

- one component: `paint-button`
- one canonical schema file: `system.schema.json`
- one authored web projection file: `system.web.config.json`
- one shared web runtime IR: `generated/system.web.json`
- one design-tool-neutral catalog IR: `generated/system.catalog.json`
- one generated adapter: `src/generated/system-web.ts`
- one generated catalog adapter: `src/generated/system-catalog.ts`
- one split model layer under `src/model/*.ts`
- one pure custom element consumer: `src/components/paint-button.ts`
- one Storybook workspace rooted at `.storybook/`
- one Storybook overview/docs consumer: `src/stories/runtime-overview.stories.ts`
- one component story consumer: `src/stories/paint-button.stories.ts`

## Refresh The Prototype

From the repo root:

```bash
cd examples/web-runtime-prototype
bun install
bun run generate
bun run build:demo
bun run build:storybook
bun run test
```

`bun run generate` does five things:

1. runs Paint twice to refresh the real backend artifacts under `generated/paint/`
2. compiles `system.schema.json` plus `system.web.config.json` into `generated/system.web.json`
3. compiles `system.schema.json` plus generated Paint artifacts into `generated/system.catalog.json`
4. generates the typed web adapter in `src/generated/system-web.ts`
5. generates the typed catalog adapter in `src/generated/system-catalog.ts`

`bun run build:demo` bundles:

- `demo/boot.ts` into `demo/dist/boot.js`
- `demo/main.ts` into `demo/dist/main.js`

`bun run serve:demo` builds that bundle and serves the example at an explicit `/demo/` mount so the
browser keeps the right base URL for relative module imports:

- `/demo/` -> `demo/index.html`
- the bundled browser bootstrap and runtime entrypoints
- the generated Paint CSS artifacts
- the shared web runtime outputs

`bun run storybook` starts a real Storybook workspace over the shared IR and generated artifacts.

`bun run build:storybook` builds a static Storybook/docs site under `storybook-static/`.

## What Is Real Versus Authored

Generated by Paint:

- `generated/paint/css/**`
- `generated/paint/web/**`
- including machine-readable `diagnostics.pack.json` projections for the live consumer surface

Generated from the canonical schema:

- `generated/system.web.json`
- `generated/system.catalog.json`
- `src/generated/system-web.ts`
- `src/generated/system-catalog.ts`

Authored in the design-system layer:

- `system.schema.json`
- `system.web.config.json`
- `src/model/runtime.ts`
- `src/model/artifacts.ts`
- `src/model/tokens.ts`
- `src/model/showcase.ts`
- `src/model/storybook.ts`
- `src/model/findings.ts`
- `src/components/paint-button.ts`
- `.storybook/main.ts`
- `.storybook/preview.ts`
- `src/stories/runtime-overview.stories.ts`
- `src/stories/paint-button.stories.ts`
- `demo/index.html`
- `demo/boot.ts`
- `demo/main.ts`

This is the point of the prototype: the custom element and the Storybook consumer stay authored,
but they do not become the source of truth.

The sibling read-only design-tool consumer lives in
[`examples/design-tool-bridge-prototype/README.md`](../design-tool-bridge-prototype/README.md).
